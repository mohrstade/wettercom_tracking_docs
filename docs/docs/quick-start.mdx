# Quick Start: Adding and Editing Events

This guide explains how to add a new tracking event or edit an existing one.

All events are defined as **JSON Schema files**. The documentation is auto-generated from these files — you never edit the `.mdx` files directly.

---

## How it works

```
static/schemas/next/events/
  ├── your-event.json        ← edit or add here
  └── components/
      └── dataLayer.json     ← shared base schema
```

After changing a schema, run the doc generator to update the docs:

```bash
npm run gen-docs
```

The site auto-reloads during development (`npm run start`).

---

## Editing an existing event

1. Open the event's JSON file in `static/schemas/next/events/`, e.g. `pagetrack.json`.
2. Make your changes (add/remove a property, update a description, change `required`, etc.).
3. Regenerate the docs:

```bash
npm run gen-docs
```

4. Validate that your schema examples are still valid:

```bash
npm run validate-schemas
```

### Example: adding a new property

```json
"pageLanguage": {
  "type": "string",
  "description": "The language of the page, based on the HTML lang attribute.",
  "examples": ["de", "en"]
}
```

Add it under `"properties"` and, if mandatory, include the key in `"required"`.

## Adding a new event

### 1. Create the schema file

Create a new file in `static/schemas/next/events/`:

```
static/schemas/next/events/my-new-event.json
```

Use this minimal template:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "<SITE_URL>/schemas/events/my-new-event.json",
  "title": "my_new_event",
  "description": "Short description of what this event tracks.",
  "type": "object",
  "allOf": [
    { "$ref": "./components/dataLayer.json" }
  ],
  "required": ["$schema", "event"],
  "properties": {
    "$schema": {
      "description": "Schema reference for validation.",
      "const": "<SITE_URL>/schemas/events/my-new-event.json",
      "type": "string"
    },
    "event": {
      "type": "string",
      "const": "my_new_event",
      "description": "The GTM dataLayer event name."
    }
  }
}
```

Replace `<SITE_URL>` with your site's base URL (see `docusaurus.config.js` → `url`).

### 2. Register the event in `event-reference.json`

Open `static/schemas/next/event-reference.json` and add a `$ref` to your new event inside `"oneOf"`:

```json
{
  "oneOf": [
    { "$ref": "./events/pagetrack.json" },
    { "$ref": "./events/radar.json" },
    { "$ref": "./events/my-new-event.json" }
  ]
}
```

### 3. Generate the docs

```bash
npm run gen-docs
```

This creates the `.mdx` page automatically and adds the event to the Event Reference index.

### 4. Validate

```bash
npm run validate-schemas
```

Fixes any issues flagged here before committing.

## Property naming rules

All property names (except `$schema`) must follow the `dataLayer.json` naming convention:

- Lowercase letters, numbers, and underscores only: `^[a-z0-9_]+$`
- Max 40 characters

---

## Releasing a new version

Once your `next` schemas are ready to publish as a stable version:

```bash
npm run version 1.3.0
```

This command:
1. Snapshots the current `docs/` into `versioned_docs/version-1.3.0/`
2. Copies `static/schemas/next/` to `static/schemas/1.3.0/`
3. Updates all `$id` URLs in the versioned schemas
4. Regenerates docs for the new version

Commit and push all generated files afterwards.
